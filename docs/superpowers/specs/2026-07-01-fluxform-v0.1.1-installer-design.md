# FluxForm v0.1.1 Installer Design

## Goal

FluxForm v0.1.1 changes the primary desktop distribution from a direct-run WPF publish folder to a standard Windows installer. The installer should make the app feel safe for ordinary users by placing runtime files in a fixed application directory and exposing only normal shortcuts.

## Scope

v0.1.1 covers the WPF desktop app only. The CLI publish output remains available for development and advanced users, but it is not added to the installer or exposed through shortcuts in this release.

The installer must:

- Default to installing FluxForm under `%LocalAppData%\Programs\FluxForm`.
- Allow the user to choose a different installation directory.
- Include `FluxForm.WPF.exe` and the bundled `tools/ffmpeg` runtime files.
- Create a Start Menu shortcut.
- Offer a desktop shortcut during installation.
- Provide a normal uninstall entry.
- Optionally launch FluxForm after installation.
- Support English and Simplified Chinese installer UI text.

The installer must not:

- Add file associations.
- Add Explorer context menus.
- Modify `PATH`.
- Install or download LibreOffice.
- Add automatic updates.
- Change conversion behavior.

## Packaging Approach

Use Inno Setup to build a standard installer executable named `FluxFormSetup-0.1.1.exe`.

Configure the installer as a per-user installation by default. It should not require administrator privileges for the normal install path. The default directory should be equivalent to `{localappdata}\Programs\FluxForm`, and the directory selection page should remain available so users can choose another location.

Configure installer languages for English and Simplified Chinese. The installer should default to the user's Windows language when Inno Setup can determine it, while still allowing the user to choose another supported language.

Add:

- `installer/FluxForm.iss` for the Inno Setup definition.
- `scripts/publish-installer.ps1` to publish the WPF app and invoke the Inno Setup compiler.

The installer build script should use the existing WPF publish flow with FFmpeg bundling enabled, then compile the installer into `publish/installer`.

`scripts/publish-installer.ps1` should accept an optional `-InnoSetupCompilerPath` parameter. If the compiler cannot be found from that parameter or common installation paths, the script should fail with a clear message that tells the developer to install Inno Setup or pass the compiler path explicitly.

## Windows Security and Signing

v0.1.1 can generate an unsigned installer, but the release process must treat SmartScreen warnings as expected until the project has code signing in place and enough download reputation.

Microsoft Defender SmartScreen can warn when a downloaded app or installer is not known, not commonly downloaded, or lacks established reputation. Code signing reduces this risk because reputation can attach to the signing certificate, but signing does not guarantee immediate warning-free installs for a new certificate.

For v0.1.1:

- The installer script should not require a signing certificate.
- README or release notes should mention that early unsigned builds may show a Windows SmartScreen warning.
- The implementation should leave room for a future signing step, but signing itself is out of scope for v0.1.1 unless a certificate is already available.

## Versioning

Update shared project version metadata from `0.1.0` to `0.1.1` in `Directory.Build.props`.

Update tests that assert the declared application version so CI reflects the intended release.

## Documentation

Update README and developer instructions so normal users are directed to the installer. Keep direct publish folder usage documented as a development or advanced distribution path.

Document that LibreOffice remains an external dependency for document conversion.

## Release Check Integration

Keep `scripts/release-check.ps1` usable on machines that do not have Inno Setup. Installer generation should be optional at this stage, for example behind a `-BuildInstaller` switch.

This prevents ordinary build/test validation from failing on developer machines or CI runners that are not prepared for installer creation.

## Acceptance Criteria

- `Directory.Build.props` declares version `0.1.1`.
- The WPF app can still be published by the existing script.
- `scripts/publish-installer.ps1` can generate `publish/installer/FluxFormSetup-0.1.1.exe` when Inno Setup is available.
- The installer installs the WPF app under a current-user application directory.
- The installer defaults to `%LocalAppData%\Programs\FluxForm`.
- The installer allows the user to choose another installation directory.
- The installer supports English and Simplified Chinese UI language selection.
- Start Menu shortcut launches FluxForm.
- Desktop shortcut is optional.
- Uninstall removes the installed application files.
- Documentation explains that unsigned early installers may trigger Windows SmartScreen.
- README describes the installer as the normal user entry point.
- Tests cover the new version and installer script/configuration expectations.
